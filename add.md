# Agent 项目设计文档

## 一、项目概述

在 MALL 电商平台基础上增加两个 Agent 的设计：

### 1.1 客服 Agent（ChatService）

- **定位**：处理重复、标准问题的自动回复
- **架构**：1 SupervisorAgent + 4 ReactAgent（售前/售后/转人工/闲聊）
- **知识库**：150 条商品 FAQ，Python 脚本从 MySQL 生成，存入 Milvus `faq` collection
- **检索方式**：FAQ 向量检索（searchFaq）+ 规则精确查询（ReturnRuleKnowledgeTools）
- **多轮对话**：ConversationState 结构化状态 + 分层记忆（近期 3 轮原文 + 老轮次压缩摘要）
- **上下文绑定**：每个会话 conversationId 与商品 Id/用户 Id 绑定，指代词自动解析
- **兜底**：知识库无匹配 → 转人工（escalation_agent）

### 1.2 退货审核 Agent（ReturnAgentService）

- **定位**：自动审核退货申请，结合商品属性/收货时长/退货原因/用户历史综合判定
- **架构**：单 ReactAgent + 7 工具 + 判定框架 prompt
- **知识库**：`rules.json`（4 条自动通过规则 + 20+ 升级关键词 + 6 类严格拒绝品类）
- **检索方式**：`analyzeReturnText`（混合检索 + 精确匹配封装，Agent 不感知底层检索策略）
- **硬门禁**：同一用户连续拒绝 ≥3 次 → 自动转人工
- **优化**：LRU 缓存（184x 加速）+ 工具输出精简（~70% token 节省）

### 1.3 客服-退货联动（暂未实现）

- 客服界面嵌入退货入口
- 识别退货诉求 → 触发退货 API → 自动关联商品 Id 和订单 Id

## 二、知识库体系（两套独立）

### 2.1 FAQ 知识库

| 维度 | 说明 |
|------|------|
| 存储 | Milvus `faq` collection (HNSW, L2, 1024 维) |
| 数据 | 150 条 Q&A，Python 脚本自动生成 |
| 检索方式 | 纯向量语义（`searchFaq`，DashScope text-embedding-v4） |
| 适用场景 | 通用政策问题（运费/快递/保修），口语化问题（"能用微信付吗"→"支持哪些支付方式"） |

### 2.2 退货规则知识库

| 维度 | 说明 |
|------|------|
| 结构化规则 | `rules.json`（含 synonyms + examples，支持同义词扩展和编辑距离） |
| 向量索引 | Milvus `rule_keywords` collection (HNSW, L2, 1024 维) |
| 精确索引 | 内存 HashMap（启动时从 `rules.json` 构建三层漏斗） |
| 检索方式 | 混合检索（向量路 + 精确路双路并行 → 加权合并去重） |

## 三、客服 Agent 详解

### 3.1 架构

```
SupervisorAgent "agent_router"
  ├── pre_sales_agent  (售前) ── FaqTools（商品咨询/价格/对比/推荐）
  ├── post_sales_agent (售后) ── FaqTools + ReturnRuleKnowledgeTools（退货资格诊断）
  ├── escalation_agent (升级) ── 零工具（纯话术格式化 + 情绪安抚）
  └── chitchat_agent   (闲聊) ── DateTimeTools（问候/感谢/无关话题）
```

### 3.2 路由策略

- 商品/价格/对比/推荐 → `pre_sales_agent`
- 退货/退款/物流/保修 → `post_sales_agent`
- 投诉/律师/法律/不确定 → `escalation_agent`
- 问候/感谢/无关话题 → `chitchat_agent`

### 3.3 多轮对话

- 状态管理：`ConversationState`（意图/实体/澄清问题/轮次）
- 分层记忆：近期 3 轮原文（~200 tokens）+ 老轮次压缩摘要（~50 tokens）
- 指代消解：用户说"这个"，从 `entities` 推断指代对象
- 澄清能力：意图模糊或指代不明时，Agent 追问而非直接转人工

### 3.4 检索策略

- FAQ 查通用政策 → `searchFaq`（纯向量，Milvus HNSW）
- 退货资格判断 → `queryStrictRejectRules` + `queryAutoApproveRules`（规则库子串匹配）
- 为什么 FAQ 不用混合检索：FAQ 是长文本语义匹配，精确路几乎零命中

## 四、退货 Agent 详解

### 4.1 架构

单 ReactAgent + 7 工具 + 判定框架 prompt（4 维度：品类限制 > 退货条件 > 时间窗口 > 升级风险）

### 4.2 核心工具

```
return_audit_agent
  ├── getApplyDetail / getRejectCount     — 信息收集
  ├── analyzeReturnText                   — 语义分析（混合检索+精确匹配封装）
  ├── queryStrictRejectRules              — 规则库查询（contains 子串）
  ├── queryAutoApproveRules               — 规则全文读取
  ├── queryReceiptThresholds              — 天数阈值读取
  └── getOrderTime / calculateReceiveDays — 时间计算
```

### 4.3 检索策略

| 方式 | 用途 | 实现 |
|------|------|------|
| 混合检索 | 扫描退货文本中的规则信号 | 向量路（Milvus HNSW）+ 精确路（三层漏斗） |
| 精确匹配 | 确认实体词（12315/律师/消协） | HashMap exact→synonym→fuzzy |
| 规则读取 | 获取规则完整定义 | `rules.json` 子串 contains() |

**三者互补**：检索回答"文本里有什么关键词"，规则读取回答"这个规则的具体条件是什么"。

### 4.4 短路逻辑

```
rejectCount ≥ 3 → 直接转人工（跳过所有检查）
strict_reject 命中 → 跳过 auto_approve + 时间检查
信息足够 → 立即 FINISH，不为"完整性"调多余工具
```

## 五、RAG 检索优化

### 5.1 评估指标

| 指标 | 问题 | 不受重复影响 | 不受排名影响 |
|------|------|------------|------------|
| Hit@K | "找到了吗？" | ✓ | ✓ |
| Recall@K | "找全了吗？" | ✗ | ✓ |
| MRR@K | "第一个对的多靠前？" | ✓ | ✗ |

### 5.2 权重优化

- **方法**：差分进化 + 5 折交叉验证（目标函数：Recall@3）
- **参数（v2 验证权重）**：vw=1.31, kw=0.65, vt=0.30, vtk=6
- **工具**：`scan_weights.py` HTTP 实时扫描（3 分钟扫 9 组，无需重启）
- **触发条件**：`rules.json` 同义词有增删 → 重新扫描权重

### 5.3 Few-shot

检索层打不通的 7 个语义盲区（掉色→质量问题、色差→与描述不符、起红点→过敏等），在 Agent prompt 中加入语义映射表，推理层兜底。

## 八、设计决策

### 8.1 为什么用 Agent 而非传统规则引擎

传统智能客服基于意图分类 + 规则引擎（if-else + API 调用），对口语化、歧义、复杂语义理解不足。Agent 让 LLM 自主推理 + 工具调用，可消歧、可模糊匹配、可动态短路。

### 8.2 为什么退货审核用 ReAct 而非 Plan-Execute

工具数 ≤7、短路逻辑可 prompt 内置的场景下，Plan-Execute 的 3 Agent 编排层是过度设计。Plan-Execute 适用条件：工具数 >8 或需要真正并行执行。设计经验保存在 `rag-test/docs/plan_execute_patterns.md`。

### 8.3 为什么 FAQ 不用混合检索

FAQ 是长文本语义匹配（"能用微信付钱吗"→"支持哪些支付方式"），没有"漏掉就出事"的专有名词。精确路在 FAQ 场景几乎零命中，纯向量足够。

### 8.4 为什么保留 JSON 规则读取

检索回答"匹配了什么关键词"，规则读取回答"规则具体是什么"（完整条件/天数/凭证要求）。Agent 做决策需要规则全文，两者互补不可替代。


### 为什么 HNSW 冷启动比 IVF_FLAT 好

IVF_FLAT 是"先聚类再搜"——128 个簇中心在内存，但簇内原始向量在磁盘，首次查询触发大量 I/O，搜索不完整。
HNSW 是"图结构遍历"——整个图（节点+边）由 loadCollection 一次性加载到内存，查询时走完整图遍历，第一次和第一百次完全相同。

| | IVF_FLAT | HNSW |
|------|----------|------|
| 存储结构 | 聚类中心 + 簇内向量列表 | 图（节点 + 邻居边） |
| loadCollection 加载 | 聚类中心（小） | 整个图结构 |
| 冷启动 I/O | 簇内向量在磁盘，查询触发大量 I/O | 全在内存，无 I/O |
| 热稳定需求 | 多次查询填充页缓存 | 不需要预热 |
