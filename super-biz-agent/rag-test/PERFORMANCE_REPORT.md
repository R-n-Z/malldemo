**v3 废弃原因**：差分进化在模拟器中表现好（R@3=116.5%），但训练数据来自旧权重检索结果，模拟器的 merge 仿真与实际 Java 逻辑存在偏差。实际部署后分数阈值评估 Correctness 显著下降。当前保持 v2 经验证权重。

训练方法: 差分进化 + 5折交叉验证, 目标函数 Recall@3

## 四、权重训练经验教训

1. **训练数据必须从实际权重配置的检索结果中采集**，不能跨权重版本复用
2. **差分进化模拟器**的 merge 逻辑与 Java `HybridSearchTools` 存在偏差（主要在于去重排序的边界条件）
3. **评估器的分数阈值**（`expectedMinScore`）与权重强耦合——改变权重后阈值需重新标定
4. **推荐流程**：修改权重 → 跑全量检索 → 重新标定每条用例的 `expectedMinScore` → 重新训练

## 五、Agent 层优化

### 5.1 LRU 缓存

`ReturnRuleAgentService.analyzeReturnText()` 内置线程安全 LRU 缓存（100 条上限），相同退货文本避免重复调用 embedding API。

| 场景 | 冷查询 | 缓存命中 | 加速 |
|------|--------|---------|------|
| escalation 类 | 562ms | 3ms | **184x** |
| strict_reject 类 | 190ms | 3ms | **58x** |
| auto_approve 类 | 190ms | 3ms | **63x** |

### 5.2 工具输出精简

`analyzeReturnText` 输出从完整分组 JSON（~600 chars）压缩为精简格式（~150-240 chars）：

```json
{"escalation":[{"k":"投诉","s":0.383}],"hints":["升级词命中，需语境判断"]}
```

| 查询类型 | 旧输出 | 新输出 | 节省 |
|----------|--------|--------|------|
| escalation | ~600 chars | 128 chars | **79%** |
| strict_reject | ~650 chars | 242 chars | **63%** |
| auto_approve | ~750 chars | 232 chars | **69%** |

每次审核减少 ~400 chars token 消耗。

### 5.3 稳定性

5 条用例 × 3 轮测试：关键词 100% 稳定，分数范围 0.000（确定性）。HNSW 索引 + DashScope embedding 组合输出完全一致。

### 5.4 权重运行时覆盖

`/rag-test/hybrid/search` 支持 `?vw=&kw=&vt=&vtk=` 参数覆盖当前配置值，`scan_weights.py` 通过 HTTP 扫描最优权重（3 分钟扫 9 组，无需重启）。

### 5.5 性能指标采集

`ReturnAgentService` 每次审核后输出结构化指标：延迟、工具调用估算、短路触发、few-shot 触发。配套 `analyze_agent_metrics.py` 分析脚本。
